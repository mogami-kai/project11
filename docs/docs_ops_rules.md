# Ops Rules (v5)

## 1. Idempotency
- 提出系は全て `idempotencyKey` を受け付ける
- 再送は重複作成しない
- 重複時は同一結果の再返却が原則（ユーザー体験を壊さない）

---

## 2. Staff Status

| 状態 | 配信 | 提出 |
|------|------|------|
| `isActive=false` | 対象外 | 拒否して管理者誘導（v5推奨） |
| `lineFollowStatus=unfollow` | 対象外 | 提出はOK（ただし配信は飛ばさない） |

- unfollow webhook を受けて状態更新し、送信失敗を減らす

---

## 3. Hotel Operations
- `hotel.push` は対象月/現場/在籍/フォロー状態で配信対象を絞る
- 必要/不要の回答は必ずログに残す
- 予約スクショOCRは**画像保存しない**
- 不一致は必ず `ADMIN_ALERTS` に残し、後処理できる

---

## 4. Images（経費立替 領収書）
- 高画質原本をDBに保存しない
- Worker 受信時にリサイズ処理を行ってからストレージへアップロード
- リサイズ仕様は [v5_spec.md §2.2](./v5_spec.md) に準拠

```
入力  : JPEG / PNG / HEIC / WEBP（any）
出力  : JPEG, 最大1200px, 品質70%
保存  : R2 or GCS（DBにはURLのみ）
```

---

## 5. No Silent Failure
- `ops.log` / `ADMIN_ALERTS` などの監査・アラート記録が失敗したら：
  - レスポンスに `warning` を返す
  - 重要処理は `partial` / `fail` を固定

---

## 6. Admin IP Allowlist（必須設定）

### 6.1 仕様定義（Normative）

`ADMIN_ALLOWED_IPS` は **本番環境における必須設定値** である。

- Worker の以下のルートは、IP 許可リストによる保護を前提として設計されている：
  - `GET /api/admin/shift/raw/recent`（管理診断ルート）
  - `POST /api/hotel/push`（ホテル要否プッシュ）
  - `POST /api/reminder/push`（リマインダープッシュ）
- `ADMIN_ALLOWED_IPS` が未設定・空文字・不正値（空白のみ等）の状態での本番デプロイは **仕様上の禁止事項（デプロイ不可）** とする。
- 「コードが fail-close か fail-open か」は実装の詳細であり、本仕様は「値が必ず存在する」ことを前提として設計の正当性を成立させる。したがって、fail-open / fail-close いずれの実装であっても、`ADMIN_ALLOWED_IPS` が設定されている限り機能的差異は生じない。
- **例外**：ローカル開発環境では未設定を許容する。ただしその場合、上記ルートへのアクセスが意図せず許可される可能性を開発者が認識したうえで使用すること。

### 6.2 対象の環境変数

| 変数名 | 種別 | 必須 | 説明 |
|--------|------|------|------|
| `ADMIN_ALLOWED_IPS` | Worker 環境変数 | **本番必須** | カンマ区切りの許可 IP アドレス一覧。Worker 側の環境変数として設定する |

### 6.3 値の要件

- 形式：IPv4 アドレスのカンマ区切りリスト
- 空文字列・空白のみは「未設定」と同義とし、デプロイ禁止条件に該当する
- 設定値が存在するかどうかの確認は、デプロイ前チェックの一部として必ず実施する

### 6.4 デプロイ前確認事項

本番デプロイの前に、以下を確認すること：

1. `ADMIN_ALLOWED_IPS` が Worker 環境に設定されていること（空でないこと）
2. 設定値に最低 1 件以上の有効な IP アドレスが含まれること
3. 上記が確認できない場合、デプロイを中止する

確認結果は監査ログ（デプロイ記録）に「ADMIN_ALLOWED_IPS 設定済み」として残すこと。

### 6.5 運用責務

- `ADMIN_ALLOWED_IPS` の管理責任者は、本番 Worker 環境を管理する運用担当者とする
- 許可 IP の追加・削除は、Worker 環境変数の更新として扱う
- 更新後は対象ルートへのアクセスが意図どおり制限されることを確認すること

### 6.6 未設定時のリスク整理

`ADMIN_ALLOWED_IPS` が未設定の状態では、以下のリスクが存在する：

- `/api/hotel/push`・`/api/reminder/push`：API キー保有者であれば任意の IP から呼び出し可能になる可能性がある
- `/api/admin/shift/raw/recent`：コード実装に依存した挙動となり、仕様上の意図と一致しない

このため、ADMIN_ALLOWED_IPS 必須設定を仕様として明記し、コード実装の差異に依存しない運用前提を確立する。
