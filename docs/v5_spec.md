# Project1 v5 Spec (Final)

## 0. Goal
LINE userId と本人情報（フルセット）を確実に紐づけ、以後の業務ログ（交通費/経費/ホテル/シフト）を「本人単位」で破綻なく運用できる状態を v5 とする。

---

## 1. User Flows (v5)

### 1.1 Registration（必須）
全員が以下の本人情報を入力し登録完了する（未完了は提出機能を解放しない）

- 氏名（漢字・フルネーム・姓と名にスペース）
- フリガナ（全角カナ・フルネーム・スペース）
- 生年月日
- 最寄駅（曖昧回避の補足許容）
- 携帯番号
- 緊急連絡先（関係/電話）
- 郵便番号
- 住所

**目的：** LINE userId ⇄ 本人情報の確実な紐づけ（本人特定の正本）

### 1.2 Rich Menu Gating
未登録（required fields 不足）の場合、以下は利用不可：

- 交通費（手押し）
- 交通費（OCR）
- 経費立替
- 状況確認
- 来月シフト提出

可能なのは「登録導線」と「登録状況確認」のみ

### 1.3 Daily Operations
- **交通費：** 手押し or OCR 下書き → 最終確認 → 提出（同一ログに保存、画像は保存しない）
- **経費立替：** 金額/用途/備考/領収書画像を提出  
  → 領収書画像は**リサイズして保存**（詳細は `2.2 Images Policy` 参照）
- **状況確認：** 今月集計/未提出確認
- **シフト提出：** 来月稼働希望を提出 → raw保存 → 解析/集計

### 1.4 Hotel Operations
- 管理者が「ホテル必要者へ一括通知」を簡単に実行
- スタッフは「必要/不要」をボタンで回答
- 管理者が予約スクショを投入した場合：
  - OCRで名前を抽出し、スタッフマスタへ名寄せして自動仕分け
  - 不一致は必ず管理者アラートに残す（握りつぶさない）

## 2. Data Policy（DBに何を残すか）

### 2.1 Master vs Transaction
- **Staff Master：** 本人情報の正本（提出ログから参照）
- **Transaction Logs：** 交通費/経費/ホテル/シフトの業務ログ（本人は lineUserId で紐づけ）

### 2.2 Images Policy

| 種別 | 保存方針 |
|------|---------|
| 交通費スクショ（OCR入力用） | **保存しない**（OCRは下書き生成のみ、原本不要） |
| ホテル予約スクショ（管理者OCR） | **保存しない**（名寄せ処理後に破棄） |
| **領収書画像（経費立替）** | **リサイズして保存**（DB/ストレージ負荷軽減のため） |

#### 領収書リサイズ仕様
```
最大幅/高さ : 1200px（長辺基準、アスペクト比維持）
JPEG品質    : 70%
フォーマット: JPEG に統一（PNG/HEIC/WEBP → 変換）
保存先      : R2 or GCS（Worker経由でアップロード）
DB保存内容  : ストレージURL のみ（バイナリはDBに入れない）
```

TODO(v5): R2/GCS のバインディング名と公開URL生成規約は未規定。実装側で環境変数により明示する。

> **判断基準：** 高画質原本が必要なのは証跡保全のためだが、業務確認用途では
> 1200px/70% で十分な視認性を確保できる。DB直接保存は避け、オブジェクトストレージに
> URLで参照する構成とする。

### 2.3 Idempotency
- 交通費/経費/ホテル処理/シフト ingest 等は idempotencyKey を持ち二重登録を防止
- 重複は「成功レスポンスの再返却」または「重複扱いの成功」として扱う（運用を止めない）

## 3. Operational Hardening（運用を120点にする）

### 3.1 Staff Master: isActive / unfollow
- `isActive`: true/false を持つ（退職/休職等の論理無効化）
- `lineFollowStatus`: follow/unfollow を記録（unfollow webhookで更新）
- 一括配信の対象は `isActive=true` かつ `follow` のみに限定

### 3.2 Hotel OCR: normalization
名寄せは正規化して比較する：
- 全角/半角、カナ/英字、大文字小文字、スペース有無を無視
- カナは濁点/半濁点・小書きも正規化対象
- 一致しない場合は必ず `ADMIN_ALERTS` に記録し、管理者が後処理できる導線を用意

### 3.3 No Silent Failure（握りつぶし禁止）
監査ログ（ops.log）やアラート記録が失敗した場合は：
- レスポンスに `auditLogged=false` / `warning` を明示
- 重要処理は `fail` か `partial` を固定（実装で統一）

---

## 4. Done Criteria（v5）
- 登録完了まで提出機能が解放されない（APIレベルで拒否できる）
- 全提出が lineUserId に紐づき、未登録は拒否
- 交通費は手押し/OCRが同一ログに入り、画像は保存しない
- **領収書画像はリサイズ（1200px/70%JPEG）してストレージ保存、DBにはURLのみ**
- ホテル必要/不要運用が回り、管理者スクショOCRで自動仕分けできる
- unfollow/isActive/名寄せ正規化が組み込まれ、運用事故が減る
- 監査ログ/アラートが握りつぶされない
